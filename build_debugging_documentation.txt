MCQ-Scanner-Checker Build Debugging Documentation
===============================================
Date: 2026-07-10
Repository: MCQ-Scanner-Checker

Overview
--------
This document records the build failures, environment issues, commands executed, and the fixes applied while rebuilding and repairing the project in the Codespace environment.

1. Initial problem
------------------
User request:
- Unzip the new version of MCQ-Scanner-Checker
- Rebuild it
- Fix issues until the project built successfully
- Commit and push the changes

Observed state:
- The repository already contained a ZIP archive named MCQ-Scanner-Checker.zip
- The project source tree had changes already present from the new version
- The initial build failed before compilation due to missing Java environment

2. Build failure 1: invalid JAVA_HOME
-------------------------------------
Symptom:
- Gradle failed with:
  JAVA_HOME is set to an invalid directory: /usr/local/sdkman/candidates/java/current

Executed commands:
- ./gradlew assembleDebug
- echo "$JAVA_HOME"
- which java
- java -version
- find /usr/lib/jvm -maxdepth 2 -type f -name java 2>/dev/null | head -20

Challenge:
- The environment had an invalid JAVA_HOME path
- No working Java runtime was available on PATH

Fix applied:
- Installed OpenJDK 17 in the container
- Set JAVA_HOME to /usr/lib/jvm/java-17-openjdk
- Exported PATH to include the new JDK bin directory

3. Build failure 2: Android SDK location missing
------------------------------------------------
Symptom:
- Gradle failed with:
  SDK location not found. Define a valid SDK location with ANDROID_HOME or sdk.dir

Executed commands:
- ./gradlew assembleDebug
- find /usr/local/sdkman -maxdepth 4 -type d 2>/dev/null | head -100
- find / -maxdepth 4 -type d \( -name 'android-sdk' -o -name 'sdk' -o -name 'platforms' -o -name 'build-tools' \) 2>/dev/null | head -200
- which sdkmanager
- which adb

Challenge:
- The project expected sdk.dir=/opt/android-sdk
- The Android SDK was not installed in the container

Fix applied:
- Downloaded Android command-line tools
- Installed the SDK under /opt/android-sdk
- Updated local.properties to use sdk.dir=/opt/android-sdk
- Exported ANDROID_HOME and ANDROID_SDK_ROOT

4. Build failure 3: Android licenses not accepted
-------------------------------------------------
Symptom:
- Gradle reported that Android SDK platform and build tools licenses had not been accepted

Executed commands:
- yes | /opt/android-sdk/cmdline-tools/bin/sdkmanager --sdk_root=/opt/android-sdk --licenses
- /opt/android-sdk/cmdline-tools/bin/sdkmanager --sdk_root=/opt/android-sdk "platform-tools" "platforms;android-34" "build-tools;34.0.0"

Challenge:
- The Android SDK packages were downloaded but license acceptance was required before compilation could proceed

Fix applied:
- Accepted SDK licenses non-interactively
- Installed required packages: platform-tools, platforms;android-34, build-tools;34.0.0

5. Build failure 4: SDK directory permissions
---------------------------------------------
Symptom:
- Gradle failed because the SDK directory was not writable
- Error stated:
  The SDK directory is not writable (/opt/android-sdk)

Executed commands:
- chown -R vscode:vscode /opt/android-sdk
- chmod -R u+rwX /opt/android-sdk

Challenge:
- The current user did not have write access to the installed Android SDK folders

Fix applied:
- Corrected ownership and permissions so Gradle and sdkmanager could install and update packages successfully

6. Build failure 5: AAPT2 daemon startup failed
-----------------------------------------------
Symptom:
- Resource processing failed with AAPT2 startup errors, including:
  AAPT2 aapt2-8.5.2-11315950-linux Daemon #0: Daemon startup failed

Executed commands:
- find ~/.gradle/caches -path '*aapt2*linux*' -name aapt2 | head -1
- ldd <aapt2-binary>
- <aapt2-binary> version

Challenge:
- The AAPT2 binary was failing to start inside the Alpine-based container because glibc compatibility libraries were missing

Fix applied:
- Installed glibc compatibility runtime packages:
  - gcompat
  - libc6-compat
  - libstdc++

7. Build failure 6: Kotlin compile errors after environment was fixed
--------------------------------------------------------------------
Symptom:
- Gradle progressed to Kotlin compilation and failed with API mismatch errors in the new version of the app

Observed errors:
- CornerSelectActivity.kt: Type mismatch: inferred type is Int but Prefs.Template was expected
- Too many arguments for OMRProcessor.grade(...)

Executed commands:
- ./gradlew assembleDebug
- grep_search for OMRProcessor.grade usages

Challenge:
- One activity was still calling an older OMRProcessor API while the new implementation used a template-based signature

Fix applied:
- Updated CornerSelectActivity.kt to call:
  OMRProcessor.grade(warped, template)
- This aligned the call site with the current OMRProcessor implementation

8. Permanent Codespace fix
---------------------------
Problem:
- The same build setup had to be repeated every time a new Codespace was opened

Challenge:
- Manual Java/Android SDK installation and environment setup were not persistent across container creation

Fix applied:
- Updated the devcontainer setup flow so the environment is configured automatically on create/rebuild
- Modified .devcontainer/setup-android-sdk.sh to:
  - install prerequisites
  - download Android command-line tools
  - accept licenses non-interactively
  - install required Android SDK packages
  - persist environment variables in /etc/profile.d/android-sdk.sh
  - write local.properties for the workspace
  - set ownership so the vscode user can access the SDK

Files updated
-------------
- .devcontainer/setup-android-sdk.sh
- .devcontainer/devcontainer.json
- app/src/main/java/com/sirjpagdi/mcqscanner/CornerSelectActivity.kt
- chat_transcript.txt
- chat_transcript_full.txt
